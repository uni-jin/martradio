# 마트 프로모션 문자 → 방송 대본 (ChatGPT API) 연동 가이드

## 1. 파인튜닝 + API가 “맞는지”에 대한 결론

- **반드시 파인튜닝만이 정답은 아닙니다.**  
  먼저 **강한 시스템 프롬프트 + 출력 형식 고정(JSON 등) + 예시 몇 개(퓨샷)**만으로도 “마트 방송체”에 꽤 가깝게 맞출 수 있습니다.
- **파인튜닝을 고려할 만한 경우**는 아래에 가깝습니다.
  - 마트 방송에 쓰는 **말투·호칭·길이·금지어·절차(인사→본문→마무리)**를 매우 일정하게 유지해야 함
  - 실제 방송문 **데이터가 충분히** 쌓였고, 그걸로 학습 세트를 만들 수 있음
  - 같은 입력(프로모션 문자)에 대해 **반복적으로 원하는 스타일 편차가 작아야** 함
- **파인튜닝 없이도 충분한 경우**는 아래에 가깝습니다.
  - 프로모션 문자 형식이 제각각이라 **추론·요약 능력**이 더 중요함
  - 자주 바뀌는 행사/규정 반영이 잦아 **프롬프트·RAG(내부 가이드 문서)**로 빠르게 바꾸는 편이 유리함

**권장 접근:**  
1단계는 **베이스 모델 + API + 엄격한 프롬프트/스키마**로 빠르게 붙이고, 품질·일관성 목표를 수치로 잡은 뒤 미달이면 **파인튜닝(또는 소량 파인튜닝)으로 보강**하는 순서가 구현·운영 모두에서 깔끔합니다.

## 2. 구현 프로세스 (단계별)

### 0) 요구사항 고정

- 입력: **프로모션 문자 원문(복붙)** + (선택) **톤 옵션**(정석/재미/사투리 등)
- 출력: **방송용 완성 대본 1개**(또는 초안 + 사용자가 편집)
- 제약: 길이(초/글자 수), **금지 표현**, **사실 왜곡 방지**(가격·기간·수량은 원문에 없으면 지어내지 않기)
- 실패 시: 재시도, “원문만 읽고 불확실하면 ‘확인 필요’ 문구 삽입” 등 정책

### 1) 데이터·품질 기준 준비 (파인튜닝을 할지 말지의 분기점)

- **골든 세트** 구축: (원문 문자) → (이상적인 방송 대본) 쌍을 N건
- **평가 지표** 정의: 예) 금지어 위반 0건, 가격 불일치 0건, 목표 길이 범위 충족률, 내부 청취 평가 점수
- 파인튜닝을 한다면: OpenAI 권장 형식에 맞춘 **학습/검증 분리**, 개인정보·매장명 등 **마스킹 규칙**

### 2) API 설계 (백엔드 중심)

- **클라이언트에 API 키를 두지 않음.** 서버 Route Handler 등에서만 호출
- 엔드포인트 예: `POST /api/broadcast/promo-to-script`
  - Body: `{ rawText: string }`
  - Response: `{ script: string, warnings?: string[] }` (필요 시 `candidates` 배열)
- **타임아웃·재시도·레이트 리밋·로깅(원문 전문 저장 여부는 GDPR/개인정보 정책에 따라 결정)**

### 3) 모델 호출 레이어

- 공통 모듈: `buildSystemPrompt()`, `buildUserMessage()`, `parseModelOutput()`
- 출력은 가능하면 **구조화(JSON 스키마)**로 받아 파싱 실패 시 재요청
- 프롬프트 본문·규칙은 **관리자 화면의 템플릿**(`{{RAW_TEXT}}` 치환)으로 관리

### 4) 파인튜닝을 택하는 경우의 추가 단계

- OpenAI 대시보드에서 **파인튜닝 작업 생성** → 학습 완료 후 **모델 ID** 발급
- 배포 환경 변수: `OPENAI_MODEL=ft:...` 형태로 **모델만 교체** 가능하게 구성
- **베이스 모델 대비 회귀 테스트**: 골든 세트로 자동 비교 스크립트(선택)

### 5) 프론트엔드 UX

- 기존 “방송 내용” 입력란: **원문 붙여넣기 영역** + **「방송문 만들기」**
- 생성 결과를 **편집 가능한 텍스트 영역**에 채운 뒤, 사용자가 확인·수정
- 이후는 **기존 플로우 그대로**(저장 → TTS/방송 생성 등)

### 6) 검증·배포

- 스테이징에서 샘플 문자 세트로 **회귀 테스트**
- 비용·지연시간 모니터링, 에러율 알림
- 문서화: 프롬프트/모델 ID/금지어 목록 변경 시 **버전 태그** 또는 변경 이력

---

## 3. 한 줄 요약

**“파인튜닝한 모델을 API로 붙인다”는 방향이 틀리지는 않지만, 먼저 API·프롬프트·스키마로 품질을 검증한 뒤 필요할 때 파인튜닝으로 고정하는 흐름이 가장 정석적이고 운영하기 쉽습니다.**
